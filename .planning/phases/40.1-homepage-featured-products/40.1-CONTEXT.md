# Phase 40.1: Homepage Featured Products - Context

**Gathered:** 2026-06-25
**Status:** Ready for planning

<domain>
## Phase Boundary

The homepage "Featured Pieces" grid (`home.ejs` `#tk-prod-grid`) shows **real, admin-curated products** with correct dual-currency prices that re-price on currency change — replacing the hardcoded demo `PRODUCTS` array and `CURRENCY='ILS'` in `frontend/js/homepage.js`.

Delivers four things:
1. An `isFeatured` flag (+ a new `featuredOrder` field) on the Product schema, toggleable from the admin dashboard.
2. An SSR-rendered featured grid with dual `usd_price`/`ils_price` values embedded in `data-*` attributes (mirroring the `category.ejs` / `ssrDynamic.js` pattern).
3. `homepage.js` reading products + dual prices from the SSR DOM (demo data removed), rendering the active currency from `localStorage.currency` on load.
4. Live re-pricing of the grid **and** the homepage cart drawer on `currency-changed`, consistent with category/cart pages.

**Out of scope (belongs to other phases):**
- Promoting the homepage demo cart drawer to the real `model.js` cart → **Phase 43**.
- Any drag-to-reorder "manage featured" admin panel → future phase if wanted.
</domain>

<decisions>
## Implementation Decisions

### Featured selection & ordering
- **D-01:** Add a dedicated **global `featuredOrder` number field** to the Product schema; the homepage grid sorts by it. `displayOrder` is NOT reused — it is per-category scoped and meaningless for a cross-category homepage grid.
- **D-02:** Grid is **capped at 8** products (first 8 by `featuredOrder`). Admins may feature more; only the top 8 render. (Prototype layout is 4×2 on desktop.)
- **D-03:** SSR query **skips unavailable / out-of-stock** products — mirror the category query filter `available: { $ne: false }, quantity: { $gt: 0 }`. The grid simply renders fewer cards rather than showing un-buyable items.

### Empty / fallback state
- **D-04:** When **zero** featured products resolve (none flagged, or all out of stock), the SSR **omits the entire "Featured Pieces" band** (eyebrow + heading + grid + "View All Jewelry" button) — no empty hole, no fallback to newest, no placeholder message.
- **D-05:** Partial counts (1–7 featured) just render whatever exists — the grid handles fewer than 8 cards naturally.

### Card behavior & badges
- **D-06:** Cards **link to the product detail page**. Confirmed product URL pattern: `/{lang}/product/{slug}` (`backend/index.js:601` → `renderProductPage`). Use the language-aware path (`en`/`he`).
- **D-07:** Badges are **auto 'Sale' from discount only** — show a "Sale" badge when the product is discounted (`original_*_price` > current price / `discount_percentage > 0`). Drop the demo's cosmetic "Handmade" badge. No admin badge effort.
- **D-08:** **Add-to-Cart keeps the existing homepage demo drawer**, but cart items must carry **both `usd`/`ils` prices** so the drawer re-prices on `currency-changed` (satisfies Success Criterion 4). Full real `model.js` cart wiring stays in **Phase 43** — do NOT pull it forward.

### Admin toggle UX
- **D-09:** In `admin/BisliView.js` add/edit product forms: a **"Featured on homepage" checkbox**; when checked, a **`featuredOrder` number input** appears. `featuredOrder` **auto-defaults to end-of-list** (append, analogous to the `displayOrder` pre-save hook) so it is optional, not mandatory.
- **D-10:** Add a **soft, non-blocking note** in the admin when more than 8 products are featured (e.g. "X products featured — only the first 8 appear on the homepage"). No hard limit.

### Claude's Discretion
- Exact `data-*` attribute naming/rounding on featured cards — follow the established `category.ejs:103` convention (`data-usd-price`, `data-ils-price`, rounded via `Math.round`).
- Whether `featuredOrder` gets its own pre-save hook or is set inline; whether to add a Mongoose index for the featured query.
- The precise DOM shape `homepage.js` reads (dataset vs JSON blob) — pick whatever cleanly mirrors how `categoriesView.js` parses SSR cards.
- Card image source field selection from the unified `images` array (with legacy `mainImage` fallback).
</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`category.ejs` data-* card pattern + `extractProductsFromDOM()`** — directly reusable shape for embedding featured products in SSR and reading them client-side.
- **`ssrDynamic.js` query/select/sort/lean pipeline** — copy the filter (`available`/`quantity`) and `.select()` field list for the featured query (add a `featuredOrder` sort + `.limit(8)`).
- **`Product.js` `displayOrder` pre-save hook** — template for auto-assigning `featuredOrder` (find max + increment).
- **Phase 40 `currency-changed` subscription model** — the homepage grid/drawer subscribes the same way category/cart pages do; no new currency plumbing.

### Established Patterns
- **SSR + client dual-render**: `homepage.js` must read from the SSR DOM, NOT overwrite it with fresh demo data (the demo currently injects into an empty placeholder; the new version hydrates real SSR cards). Keep CLAUDE.md's dual-render rule in mind.
- **Dual-price `data-*` (rounded) attributes** drive client re-pricing — same as category/cart.
- **No `crossorigin="anonymous"`** on product images (CLAUDE.md — breaks local CDN images).
- **Cache invalidation**: featuring/unfeaturing a product should invalidate the homepage cache (`backend/cache/invalidation.js`) — homepage is a cached public page.

### Integration Points
- Home route (`ssr.js`) ↔ new featured query ↔ `home.ejs` grid ↔ `homepage.js` hydration ↔ Phase 40 `currency-changed` event.
- Admin form (`BisliView.js`) ↔ Product schema (`isFeatured`/`featuredOrder`) ↔ featured query ↔ homepage cache invalidation.
</code_context>
