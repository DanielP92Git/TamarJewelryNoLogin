# Phase 40.1: Code Review Report

**Reviewed:** 2026-06-25
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found

## Summary

Phase 40.1 adds a homepage "Featured Pieces" grid: two new `Product` fields
(`isFeatured` / `featuredOrder`) with pre-save auto-ordering, write-path coercion
in the add/update routes, a dedicated `invalidateHomePage()` cache helper, an SSR
featured query + grid in `home.ejs`, admin featuring UX in `BisliView.js`, and
client-side price hydration in `homepage.js`.

The schema, pre-save hooks, NaN-safe coercion, and the SSR query (correct
`select`, `.limit(8)`, stock filter, and graceful empty-list fallback) are
implemented carefully and match the documented decisions. However, the cache
invalidation that the whole feature depends on was wired into the add/update
paths but **not** into the delete path — so a deleted featured product keeps
appearing on the cached homepage with a now-broken product link. There are also
two i18n correctness gaps (English-only "Sale" badge and English cart names in
Hebrew) and an `innerHTML`-based cart render that does not escape product
content.

## Critical Issues

### CR-01: Deleting a featured product does not invalidate the homepage cache (stale data + broken link)

**File:** `backend/routes/products.js:1089-1095` (`/removeproduct`)
**Issue:** `invalidateHomePage()` was added to `/addproduct` (line 364) and
`/updateproduct/:id` (line 702), but the delete path only calls
`invalidateProduct` / `invalidateCategory`. The homepage cache (1-hour TTL) is
keyed independently (`/en:en:USD`, etc.), so after an admin deletes a product
that is currently in the Featured grid, the cached homepage continues to render
that product for up to an hour. The SSR card links to
`/{lang}/product/{slug}` (and `homepage.js` navigates there on click), so users
get a **404 on a deleted product**. The same gap exists in the legacy JSON
`/updateproduct` route (line 873), which performs no home invalidation even
though it can change `quantity` (the featured grid filters `quantity > 0`).
Because the add/update paths *were* updated in this same phase, the delete
omission is an inconsistent, in-scope oversight rather than pre-existing
behavior.
**Fix:** Invalidate the home page whenever a product that could be featured is
removed (cheap and safe to call unconditionally, matching the add/update paths):
```js
// in /removeproduct, after computing urlCategory / invalidating product+category:
const urlCategory = DB_TO_URL_CATEGORY[deleted.category];
if (deleted.slug && urlCategory) {
  invalidateProduct(deleted.slug, urlCategory);
} else if (urlCategory) {
  invalidateCategory(urlCategory);
}
invalidateHomePage(); // Phase 40.1: deleted product may be in the featured grid
```
Also consider calling `invalidateHomePage()` from the legacy `/updateproduct`
route, and (broader, outside this file) from the order/checkout stock-decrement
path — otherwise a featured item that *sells out* still lingers on the cached
homepage for up to the TTL, defeating the D-03 "skip out-of-stock" intent.

## Warnings

### WR-01: Cart drawer renders product name/image via `innerHTML` without escaping

**File:** `frontend/js/homepage.js:78, 111-122`
**Issue:** `addToCart` stores `card.dataset.nameEn` / image `src` verbatim, and
`renderCart` injects them with `innerHTML`:
`'<span class="tk-line__name">' + it.name + '</span>'` and
`'<img src="' + it.image + '" alt="' + it.name + '" />'`. The dataset values are
the *decoded* product name (the SSR `<%= %>` escaping is undone when read via
`dataset`). A product name containing `"`, `<`, or `>` will break the markup or
inject elements/attributes. Product names are admin-controlled (not anonymous
user input), so this is not a classic external XSS, but it is an injection sink
that will corrupt the drawer on legitimate names containing quotes/angle
brackets.
**Fix:** Build the line nodes with `textContent` instead of string-concatenated
`innerHTML` (set `img.src`/`img.alt` and `span.textContent` on created
elements), or escape the values before interpolation. The grid cards themselves
are already safe because they are server-rendered with `<%=`.

### WR-02: Hebrew homepage shows English-only "Sale" badge and English cart names

**File:** `backend/views/pages/home.ejs` (Sale badge, featured card) and
`frontend/js/homepage.js:77` (cart name)
**Issue:** Two locale regressions in the featured flow:
1. The discount badge text is hardcoded `Sale` with no `lang === 'heb'` branch,
   so Hebrew shoppers see an English "Sale" label while every other string on
   the card is translated.
2. `addToCart` sets the line name to `card.dataset.nameEn || card.dataset.nameHe`
   regardless of the active language, so the cart drawer shows the English name
   even in Hebrew — inconsistent with the card title, which correctly prefers
   `name_he` in Hebrew.
**Fix:** Branch the badge text (`<% if (lang === 'eng') { %>Sale<% } else { %>מבצע<% } %>`),
and select the cart name by current language, e.g. read
`document.documentElement.lang` (or `localStorage.language`) and prefer
`nameHe` when Hebrew, mirroring the SSR `productName` logic.

### WR-03: Featured grid clickability silently depends on a populated `slug`

**File:** `backend/views/pages/home.ejs` (`data-slug="<%= product.slug %>"`) and
`frontend/js/homepage.js:169`
**Issue:** The card-click handler bails when `!card.dataset.slug`. Per CLAUDE.md,
Hebrew-only product names can produce an empty slug from the pre-save hook. Such
a product can be marked featured (the admin UX imposes no slug requirement) and
will render in the grid but be silently non-navigable — clicking does nothing,
with no fallback to the numeric `id` route. This is an easy-to-miss dead
interaction rather than a crash.
**Fix:** Guarantee a slug for featured products (validate on the admin
featuring toggle, or fall back to `/{lang}/product/{id}` when `slug` is empty in
the click handler).

## Info

### IN-01: `data-images` JSON is emitted on homepage cards but never consumed

**File:** `backend/views/pages/home.ejs` (featured card `data-images="<%= JSON.stringify(galleryImages) %>"`)
**Issue:** The full gallery-image JSON block (copied from `category.ejs`) is
serialized into every featured card, but `homepage.js` only reads price/slug/
name datasets — it never parses `data-images` (that parsing lives in
`categoriesView.js`, which does not run on the home route). The ~20 lines of
`galleryImages` construction and the per-card JSON payload are dead weight.
**Fix:** Drop `galleryImages` and the `data-images` attribute from the featured
card unless a consumer is added; keep only `desktopImage` for the `<img>`.

### IN-02: `invalidateHomePage()` fires on every product add/update, even non-featured

**File:** `backend/routes/products.js:364, 702`
**Issue:** Both write paths call `invalidateHomePage()` unconditionally, so
editing a product that is not (and never was) featured still flushes all four
home cache variants. Safe and defensive, but it erodes the cache hit rate of the
busiest page on unrelated edits.
**Fix:** Gate the call on featured involvement, e.g. only invalidate when
`product.isFeatured` is true or `req.body.isFeatured`/`featuredOrder` changed.

### IN-03: `featuredOrder` is non-unique; tie-breaking is nondeterministic

**File:** `backend/models/Product.js:93, 156-180` and `backend/routes/ssr.js:163-166`
**Issue:** Two featured products can share the same `featuredOrder` (no unique
constraint), and the auto-assign hook can also collide if multiple featured
products have `null` order. `sort({ featuredOrder: 1 }).limit(8)` then breaks
ties arbitrarily, so which 8 appear (and in what order) is not stable across
requests.
**Fix:** Add a deterministic secondary sort key (e.g. `.sort({ featuredOrder: 1, id: 1 })`)
to make ordering stable; optionally surface duplicate-order warnings in the
admin UI.

### IN-04: Redundant id comparison when counting other featured products

**File:** `admin/BisliView.js` (`editFeaturedBase` filter)
**Issue:** The exclude-self filter checks both
`String(p._id) !== String(product._id) && String(p.id) !== String(product.id)`.
Either identifier alone uniquely identifies the product; the compound condition
is redundant and would also mis-handle the (impossible-by-invariant) case where
only one identifier matches.
**Fix:** Compare a single stable identifier (`String(p._id) !== String(product._id)`).
