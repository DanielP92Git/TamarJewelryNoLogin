<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| featured card data-* → model.cart | Homepage `.tk-prod` admin-authored data attributes flow into the real cart via `window.tkAddToCart`. |
| global window event → drawer open | `cart:item-added` is a same-origin, no-payload signal that triggers a re-render + open. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-43-01 | Tampering/Injection (XSS) | featured card data → cart line | mitigate | Card data flows through `model.handleAddToCart`→`addToLocalStorage`, which stores plain fields; the drawer renders them via `textContent` (Plan 03 T-43-01). No `innerHTML` interpolation of card-sourced names. |
| T-43-09 | Spoofing | `cart:item-added` event | accept | Same-origin window event carrying no payload; worst case is an extra drawer open/re-render (no data path, no privilege). No mitigation warranted. |
| T-43-10 | Tampering | `window.tkAddToCart` global | mitigate | The hook only forwards a DOM element to the existing `model.handleAddToCart` (same path category pages already use) and dispatches a no-payload event; it adds no new privileged capability beyond what the bundled views already expose. |
</threat_model>
