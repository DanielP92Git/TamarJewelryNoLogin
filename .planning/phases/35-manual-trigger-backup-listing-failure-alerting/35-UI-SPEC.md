# Phase 35 — UI Design Contract

> Visual and interaction contract for Phase 35: Manual Trigger, Backup Listing & Failure Alerting.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase Scope Note

Phase 35 is a **backend-only phase**. It produces REST endpoints and a background service — no new HTML, CSS, or rendered components. The "UI hint: yes" in the roadmap means two things:

1. **API response shape** must be designed to serve Phase 37's admin dashboard panel (locked contract — Phase 37 must not need a breaking API change).
2. **Email alert content** is user-facing copy that must be specified precisely.

The design tokens in this contract describe the **existing admin design system** that Phase 37 will consume. Phase 35 must not deviate from these tokens in its email HTML (if HTML email is used) and must return API field names that map cleanly to the existing badge/status component semantics.

## API Response Shape Contract

This contract is locked for Phase 37 consumption. Phase 37 must not require breaking changes to these field names or types.

### POST /admin/backup — 200 Success

```json
{
  "success": true,
  "status": "success",
  "filename": "backup-2026-04-07T03-00-00Z.archive.gz",
  "sizeBytes": 8421376,
  "durationMs": 4821,
  "timestamp": "2026-04-07T03:00:00.000Z",
  "retentionDeleted": 0,
  "error": null
}
```

### POST /admin/backup — 409 Conflict (backup in progress)

```json
{
  "success": false,
  "error": "Backup already in progress"
}
```

### POST /admin/backup — 500 Failure (backup ran but failed)

```json
{
  "success": true,
  "status": "failed",
  "filename": "backup-2026-04-07T03-00-00Z.archive.gz",
  "sizeBytes": null,
  "durationMs": 1203,
  "timestamp": "2026-04-07T03:00:00.000Z",
  "retentionDeleted": 0,
  "error": "mongodump exited with code 1: connection refused"
}
```

Note: HTTP 200 is returned even when `status: "failed"` — the endpoint succeeded in running the backup service; the backup itself failed. Callers must check `result.status`, not HTTP status code alone.

### GET /admin/backups — 200 Success

```json
{
  "success": true,
  "backups": [
    {
      "filename": "backup-2026-04-07T03-00-00Z.archive.gz",
      "lastModified": "2026-04-07T03:00:04.000Z",
      "sizeBytes": 8421376,
      "status": "success",
      "durationMs": 4821,
      "error": null,
      "trigger": "cron"
    },
    {
      "filename": "backup-2026-04-06T03-00-00Z.archive.gz",
      "lastModified": null,
      "sizeBytes": null,
      "status": "failed",
      "durationMs": 1203,
      "error": "mongodump exited with code 1: connection refused",
      "trigger": "cron"
    }
  ]
}
```

Sorted: newest `lastModified` first. Failed entries (no Spaces object) sort by `BackupLog.timestamp`.

Field display hints for Phase 37:
- `filename` → strip prefix, display as date string parsed from ISO segment
- `sizeBytes` → format as "8.0 MB" (divide by 1,048,576, 1 decimal place)
- `durationMs` → format as "4.8s" (divide by 1000, 1 decimal place)
- `lastModified` / `null` → display "—" when null
- `trigger: null` → display "—" when null

## Copywriting Contract

Source: email alert spec from CONTEXT.md decisions D-03, D-07.

| Element | Copy |
|---------|------|
| Primary CTA (Phase 37 button) | "Run Backup" |
| Email alert subject | "Backup Failed — Tamar Kfir Jewelry" |
| Email alert heading | "MongoDB Backup Failed" |
| Email alert body | "A scheduled backup failed on {timestamp}. Error: {error_message}. Filename attempted: {filename}. Duration before failure: {duration_ms}ms. Check App Platform logs for details." |
| Empty state — backup list (Phase 37) | "No backups yet" / "Trigger a manual backup or wait for the scheduled run at 03:00 UTC." |
| Error state — POST /backup fails | "Backup failed: {error}. Check App Platform logs for the full error." |
| Conflict state — 409 response | "A backup is already running. Wait for it to complete before triggering another." |
| Destructive confirmation | Not applicable to Phase 35 — restore is Phase 36. |

Email alert tone: factual, no exclamation marks, include all diagnostic fields (timestamp, filename, error, duration) so the admin can diagnose without opening App Platform logs.

## Interaction States

### POST /admin/backup — State Machine

```
idle
  → [trigger] → running (isBackupRunning = true)
    → [success] → idle, return 200 { status: "success", ... }
    → [failure] → idle, return 200 { status: "failed", error: "..." }
    → [concurrent trigger] → 409 { error: "Backup already in progress" }
```

Loading state (Phase 37 button): disable button + show spinner while awaiting POST response. Synchronous response (D-16) means the request may take 10–30 seconds for a real backup. The button must remain disabled and show a loading state for the full duration.

### GET /admin/backups — Loading State

Fetch on page load. Show skeleton rows while loading. No polling — data is refreshed on manual reload or after POST /backup returns.
