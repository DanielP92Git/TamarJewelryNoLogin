## v1.5 Requirements

Requirements for Bilingual Product Content milestone. Each maps to roadmap phases.

### Schema & Migration

- [x] **SCHEMA-01**: Product schema has bilingual fields (name_en, name_he, description_en, description_he)
- [~] **SCHEMA-02**: Existing products migrated — runtime normalization provides functional equivalent (tech debt: migration script exists but not executed)
- [x] **SCHEMA-03**: Legacy name/description fields maintained during transition for backward compatibility
- [x] **SCHEMA-04**: Cart handles transition from single-language to bilingual product names gracefully

### Translation Service

- [x] **TRANS-01**: Google Cloud Translation API v3 integrated as backend service
- [x] **TRANS-02**: Backend translation endpoint (POST /admin/translate) for on-demand admin use
- [x] **TRANS-03**: Translation service caches results in memory to reduce API costs
- [x] **TRANS-04**: Translation errors handled gracefully — admin can still save with manual entry
- [x] **TRANS-05**: Bulk operations use batching with delays to respect API rate limits

### Admin Workflow

- [x] **ADMIN-01**: Admin product form shows side-by-side Hebrew and English fields for name and description
- [x] **ADMIN-02**: "Translate" button translates content between Hebrew and English on demand
- [x] **ADMIN-03**: Admin can manually edit translated text before saving
- [x] **ADMIN-04**: Product list shows translation status indicators (translated / needs translation)
- [x] **ADMIN-05**: Bulk translate tool translates all products missing a language at once
- [x] **ADMIN-06**: Bulk translate shows progress indicator and handles API rate limits

### Display

- [x] **DISP-01**: SSR product page shows correct language name/description based on URL (/en/ or /he/)
- [x] **DISP-02**: SSR category page shows correct language name/description for product cards
- [x] **DISP-03**: Client-side views show correct language with same logic as SSR
- [x] **DISP-04**: Cart displays product names in the user's current language
- [x] **DISP-05**: Graceful fallback to English when Hebrew translation is missing
- [x] **DISP-06**: JSON-LD structured data uses language-specific content with inLanguage property
- [x] **DISP-07**: OG meta tags use localized product descriptions for social sharing

### Cache

- [x] **CACHE-01**: Product update clears cached pages for both /en/ and /he/ variants
- [x] **CACHE-02**: Bulk translation triggers cache invalidation for all affected products
- [x] **CACHE-03**: Category cache cleared when products in that category are translated

## Out of Scope

| Feature | Reason |
|---------|--------|
| Real-time translation during typing | Wastes API quota, admin should write first then translate |
| Auto-translate without admin review | Machine translation always needs human review for quality |
| Locked translation fields | Admin must always be able to edit translations manually |
| Separate translation manager page | Overkill for ~94 products, inline editing sufficient |
| Language-specific slugs (/he/product/hebrew-slug) | English-only slugs simpler for routing, consistent with current approach |
| Client-side translation (browser-side API calls) | Security risk exposing API credentials, server-side is correct |
