## Gap Analysis

### SCHEMA-02: Migration Script Not Executed

**What:** The migration script (`backend/migrations/20260213000000-add-bilingual-product-fields.js`) was created in Phase 27 Plan 01 but never executed against the database.

**Functional Impact: LOW** — `normalizeProductForClient` in `backend/index.js` (lines 590-613) dynamically populates bilingual fields in every API response, so all downstream consumers (SSR, frontend, admin) work correctly.

**Why it matters:** The phase goal explicitly says "migrated" not "dynamically populated at runtime." Database products lack bilingual fields natively, creating:
- Performance overhead (field copy on every API response)
- Database doesn't reflect true bilingual state
- Direct DB queries (outside normalizeProductForClient) won't include bilingual fields

**Resolution options:**
1. Run migration: `cd backend && npm run migrate:up` (recommended)
2. Accept as tech debt and document runtime normalization as intentional strategy

### Pre-existing: cartView._render Bug

Already tracked in PROJECT.md as known tech debt. Currency-changed event handler calls `this._render()` which doesn't exist. Not introduced by v1.5 — affects currency switching on cart page only.
