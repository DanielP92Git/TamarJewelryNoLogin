<dual_render_rules>
CRITICAL — read CLAUDE.md "SSR + Client Dual-Render" and "Chrome Rollout CSS Conflicts" before editing.

For the NAV specifically (confirmed by scouting):
- The header/nav is SSR-static chrome rendered by `backend/views/partials/header.ejs`. View.js does
  NOT rewrite the header markup — `setLanguage()` only attaches behavior, updates selector text, and
  binds the hamburger / cart drawer (no innerHTML rewrite of the header). So the submenu MARKUP lives
  in ONE place: `header.ejs`. There is no JS markup twin to keep in sync for the nav.
- View.js IS the JS twin for nav BEHAVIOR. Adding the toggle logic there is the "mirror" — markup in
  header.ejs, behavior in View.js. Do not put markup in View.js and do not put behavior in the EJS.
- View.js is Parcel-bundled. After editing it you MUST run `npm run build` in `/frontend` and restart
  the backend (it reads new bundle hashes at startup). EJS and CSS changes do NOT need a rebuild.
- The submenu must survive the in-page language toggle. Because the SSR chrome is never destroyed and
  the bind uses a double-bind guard (see `_bindHamburgerMenu`), behavior persists across toggles.
</dual_render_rules>
